## What agentsview is

A **local-first viewer for AI *coding-agent* session history**. It is a single
Go binary (plus optional Tauri desktop app and Docker image) that discovers
session log files written by coding agents on the developer's machine, syncs
them into a local SQLite archive, and serves a Svelte SPA to **browse, search,
compare, and cost-track** those sessions. Tagline (README:1-4): "Browse, search,
and track costs across all your AI coding agents. One binary, no accounts,
everything local."

Target user (`PRODUCT.md:9-20`): developers running multiple AI coding agents
who need to inspect/search/audit local session history in a debugging, review,
or cost-monitoring workflow. Explicitly **no external accounts, everything
local**.

What it shows/does (README + `docs/`):

- **Session browsing** — list + detail view of every agent session, with the
  full transcript: user/assistant messages, tool calls, subagent calls,
  thinking blocks, parallel tool groups, code/diff blocks (components under
  `frontend/src/lib/components/content/`: `MessageList`, `ToolBlock`,
  `SubagentCalls`, `ThinkingBlock`, `ParallelGroup`, `CallGroup`, `CodeBlock`,
  `ActivityLane`, `SessionVitals`, `SignalPanel`).
- **Full-text search** across all transcripts (SQLite FTS5).
- **Usage / cost tracking** — token consumption and compute cost across all
  agents; positioned as a faster local replacement for `ccusage` (README:175+).
- **Analytics / trends / activity / recent-edits** dashboards.
- **AI-generated "Insights"** — local LLM summaries of sessions, generated by
  the user's own on-machine agent (`docs/insights.md`).
- **Secrets scanning** of session content (`/api/v1/secrets`).
- **MCP server** exposing sessions to other agents (`cmd/agentsview/mcp.go`).
- **Multi-machine sharing** via optional PostgreSQL push-sync, DuckDB mirror,
  Quack remote, and S3 session roots.

## Architecture & module map

Backend is **Go**; frontend is a **Svelte 5 SPA embedded into the Go binary at
build time**. Data pipeline (from `AGENTS.md` "Architecture"):

```
CLI (agentsview) -> Config -> DB (SQLite/FTS5)
                     |            |
              File Watcher -> Sync Engine -> Parsers (per agent)
                     |            |
              HTTP Server -> REST API + SSE + Embedded SPA
                                 |
                          PG Push Sync -> PostgreSQL (optional)
```

Go module `go.kenn.io/agentsview` (`go.mod:1`). Layout (`AGENTS.md`
"Project Structure", verified against tree):

- `cmd/agentsview/` — CLI entrypoint, server startup, file watcher, daemon,
  `serve`/`sync`/`usage`/`pg`/`duckdb`/`mcp`/`session`/`import`/`export`
  subcommands (Cobra). Also `managed_caddy*`, `secrets`, `doctor`, `prune`.
- `internal/config/` — config loading, JSON migration, flag registration.
- `internal/db/` — SQLite sessions/messages/search/analytics + schema
  (WAL + FTS5).
- `internal/postgres/` — optional PG push-sync, read-only store, schema.
- `internal/parser/` — one parser per supported agent; `types.go` is the
  agent registry.
- `internal/server/` — HTTP router, handlers, SSE, middleware, search, export.
  API defined via the **huma** framework in `huma_routes_*.go`.
- `internal/sync/` — sync engine, file watcher, discovery, hashing.
- `internal/web/` — embedded frontend (`frontend/dist/` copied at build).
- `frontend/` — Svelte 5 + Vite + TypeScript SPA. `src/lib/` has `api/`
  (OpenAPI-generated client), `stores/`, `components/` (activity, analytics,
  insights, usage, trends, filters, sidebar, command-palette, content, …),
  `virtual/` (virtualized lists via `@tanstack/virtual-core`).
- `desktop/` — Tauri (Rust) shell wrapping the Go binary as a sidecar.

**Data flow (main path):** agent writes session file to disk → file watcher /
periodic 15-min sync detects change → per-agent parser extracts messages, tool
calls, usage → written to SQLite (FTS5 indexed) → Go HTTP server serves REST +
SSE → Svelte SPA renders session list/detail, search, dashboards. SSE
(`/api/v1/events`) pushes real-time updates to the UI.

**Canonical REST API** (from `huma_routes_*.go`): `/api/v1/` +
`sessions`, `search`, `usage`, `activity`, `analytics`, `trends`, `insights`,
`recent-edits`, `secrets`, `import`, `push`, `sync`, `remote-sync`, `settings`,
`config`, `assets`, `openers`, `pins`, `starred`, `metadata`, `health`, plus
`events` (SSE).

## Tech stack

**Backend (Go 1.26):** Cobra CLI (`spf13/cobra`), huma v2 (typed OpenAPI HTTP
framework, `danielgtaylor/huma/v2`), SQLite via `mattn/go-sqlite3` (CGO + `fts5`
build tag), `pgx/v5` for PostgreSQL, `duckdb-go` for DuckDB mirror,
`fsnotify` file watcher, `minio-go` for S3 roots, `modelcontextprotocol/go-sdk`
for the MCP server, `klauspost/compress`, `testify`/`testcontainers` for tests.
Build requires `CGO_ENABLED=1` and `-tags fts5`.

**Frontend:** Svelte 5 + Vite + TypeScript; `@kenn-io/kit-ui` design system
(git-pinned dependency); `@lucide/svelte` icons; `marked` + `dompurify` +
`shiki` for markdown/code rendering; `@tanstack/virtual-core` virtualization;
Paraglide/inlang i18n; Playwright + Vitest tests. API client is
**OpenAPI-codegen generated** from the huma schema
(`frontend/src/lib/api/generated/`).

**Desktop:** Tauri (Rust, `desktop/src-tauri/`) with the Go binary as a sidecar.

**Distribution:** curl/PowerShell install script, Homebrew cask, GitHub
Releases, Docker image `ghcr.io/kenn-io/agentsview`.

## ACP / Virtuals specificity

**None. This project has zero relationship to Virtuals Protocol or the Agent
Commerce Protocol.** Verified by full-repo grep:

- The only `ACP` string match is an unrelated code comment
  (`internal/db/automated_test.go:238` — "ACP test adapters", generic).
- All `Virtual*` matches are `VirtualSourcePath` / `ParseVirtualSourcePath` — a
  **filesystem abstraction** for addressing one logical session inside a shared
  physical DB file (`internal/parser/virtual_source_path.go`), nothing to do
  with Virtuals Protocol.
- `wallet` matches are Linux **KWallet** (OS secret store in
  `antigravity_crypto.go`).
- No blockchain, on-chain, web3, ERC, escrow, payment, job-offering, or
  agent-commerce code anywhere.

The word "agents" here means **AI coding assistants** (Claude Code, Codex,
Cursor, Gemini, Devin, Forge, OpenCode, Aider, Amp, Warp, Zed, and ~50 others
enumerated in `internal/parser/types.go`), **not** autonomous on-chain economic
agents. This is an observability tool for *your own local coding-agent runs*.

## How close to ACP Tracer (gap analysis)

**Conceptually adjacent, domain-orthogonal.** It is a session/trace *viewer*,
which is one half of what an "ACP Tracer" wants — but the trace domain is wrong.

Overlap (what it already nails):

- Session-as-a-timeline model: parse heterogeneous run logs → normalized
  message/tool-call/usage schema → searchable store → rich transcript UI.
- Excellent trace-rendering primitives: tool calls, nested subagent calls,
  parallel groups, thinking blocks, activity lane, per-session "vitals" and
  "signal" panels — directly analogous to visualizing an ACP job's phases.
- Cost/usage aggregation, real-time SSE updates, full-text search, local-first
  no-account posture.

Gaps vs an ACP Tracer:

- **No create/run of agents.** agentsview is strictly **read-only observability**
  of sessions that *already happened* on disk. ACP Tracer's "no-code
  create/run" is entirely absent — no agent authoring, no offering config, no
  job initiation.
- **No ACP/Virtuals data model.** No jobs, offerings, phases
  (request→negotiate→transact→evaluate), memos, escrow, evaluator, wallet, or
  on-chain events. Its parsers target *local coding-agent log files*, not the
  ACP API / on-chain job lifecycle.
- **No network/registry view.** It shows your machine's sessions, not a
  marketplace of buyer/seller agents.
- **Data ingestion is file/DB-based**, not API/webhook/RPC from the ACP
  backend or chain.

`[UNVERIFIED]` Estimate: agentsview is roughly a **30-40% architectural
overlap** with ACP Tracer — strong on the "track/view a trace" rail, absent on
the "create/run on ACP" rail and on the ACP domain model. It is a reference for
*how to build the tracer's viewer*, not a drop-in ACP tracer.

## Reusable ideas

- **Single self-contained binary with embedded SPA** (Go `embed` of
  `frontend/dist/`) + optional Tauri desktop wrapper — clean, no-account,
  local-first distribution model worth copying for ATLAS tooling.
- **Typed API-first backend (huma) → OpenAPI → codegen'd frontend client** — one
  source of truth for the API contract; the ACP Tracer's job/offering schema
  could follow the same pattern.
- **Provider/parser registry pattern** (`internal/parser/types.go`, one
  `AgentDef` per source) — an ACP Tracer could model each ACP data source
  (ACP REST, on-chain events, Virtuals graph) as a pluggable provider the same
  way.
- **Trace-rendering component library** — `content/` Svelte components
  (`ToolBlock`, `SubagentCalls`, `ParallelGroup`, `ActivityLane`,
  `SessionVitals`, `SignalPanel`, `MessageList` with virtualization) map almost
  1:1 onto rendering an ACP job's phase timeline and sub-steps.
- **Real-time via SSE** (`/api/v1/events`) for live job/session updates.
- **Local SQLite + FTS5 + optional PostgreSQL push-sync** for a "local by
  default, shared when needed" storage tier — good fit for a tracer that starts
  local and later centralizes.
- **Built-in secrets scanning** of ingested content — relevant since ACP
  traces may carry sensitive payloads.
- **MCP server exposing the archive** — lets other agents query traces; ATLAS
  could expose ACP job history to its own agents the same way.

## Security notes

Strong local-first security posture (verified in `README.md` + `SECURITY.md`
present):

- Binds to **loopback only**, validates the request `Host` header to block
  **DNS-rebinding** attacks; forwarded access requires explicit `--public-url` /
  `--public-origin` (README:77-98).
- `--require-auth` for any non-loopback exposure; Docker examples publish on
  `127.0.0.1` only.
- Reads agent session dirs **read-only** (`:ro` mounts); explicitly warns not to
  paste tokens/OAuth files into bug reports and *intentionally ignores* OAuth /
  config paths under Devin roots (README:51-56).
- Secrets scanning endpoint to detect leaked credentials in transcripts.
- Quack remote requires token + TLS or explicit `--allow-insecure`.

`[UNVERIFIED]` Considerations for reuse in ATLAS: the Docker container "runs as
root" (README:113) — noted as a tradeoff, prefer named volumes. CGO/SQLite +
duckdb + many parsers is a large native attack surface if ever exposed beyond
loopback.
